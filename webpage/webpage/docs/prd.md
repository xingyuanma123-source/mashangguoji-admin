# 客服后台管理网页 - 需求文档（v1）

## 一、项目概述

本项目是一个独立的 H5 网页项目，作为\"司机报账微信小程序\"的客服管理后台。用于客服人员在电脑浏览器中管理司机提交的报账记录、车辆信息、司机信息、备用金等。

**⚠️ 关键：本项目必须连接到已有的小程序项目的 Supabase 数据库（同一个 SUPABASE_URL 和 SUPABASE_ANON_KEY），共用同一套数据，不需要重新建库。** 除了 operation_logs 表需要新建外，其他6张表已在小程序项目中创建完成。

**使用人群：** 公司客服人员（多人），通过电脑浏览器访问
**核心场景：** 司机在小程序提交报账 → 客服在网页端审核、修改、填写提成、确认 → 确认后自动计入总表 → 客服可查看和导出总表

## 二、共用数据表（已存在，不需要新建）

以下6张表已在小程序端创建，客服网页直接使用：

| 表名 | 用途 |
|------|------|
| drivers | 司机信息 |
| vehicles | 车辆信息 |
| expense_records | 报账记录（核心表 兼 总表） |
| service_staff | 客服人员（登录用） |
| advance_fund_records | 备用金充值记录 |
| fee_types | 费用类型配置 |

### 需要新建的表

#### 表7：operation_logs（操作日志表）

| 字段名 | 类型 | 必填 | 默认值 | 说明 |
|--------|------|------|--------|------|
| id | 自增主键 | 自动 | — | — |
| operator_id | 关联→service_staff表 | ✅ | — | 操作人 |
| operator_name | 文本 | ✅ | — | 操作人姓名（冗余存储方便查看） |
| action | 文本 | ✅ | — | 操作类型（见下方取值） |
| target_type | 文本 | ✅ | — | 操作对象类型：expense_record / driver / vehicle / advance_fund / fee_type / staff |
| target_id | 数字 | ✅ | — | 操作对象ID |
| detail | 文本 | 否 | 空 | 操作详情描述，如\"将提成从0修改为60\" |
| created_at | 日期时间 | 自动 | 当前时间 | — |

**action 取值：**
| 值 | 含义 |
|----|------|
| confirm | 确认报账 |
| edit | 修改记录 |
| create | 新增 |
| update | 更新 |
| delete | 删除 |

**RLS策略：** 客服可查看所有日志，可插入日志。

```sql
CREATE TABLE operation_logs (
  id BIGSERIAL PRIMARY KEY,
  operator_id BIGINT NOT NULL REFERENCES service_staff(id),
  operator_name TEXT NOT NULL,
  action TEXT NOT NULL CHECK (action IN ('confirm', 'edit', 'create', 'update', 'delete')), 
  target_type TEXT NOT NULL CHECK (target_type IN ('expense_record', 'driver', 'vehicle', 'advance_fund', 'fee_type', 'staff')), 
  target_id BIGINT NOT NULL,
  detail TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

ALTER TABLE operation_logs ENABLE ROW LEVEL SECURITY;

CREATE POLICY \"客服可以查看所有日志\" ON operation_logs
  FOR SELECT USING (true);

CREATE POLICY \"客服可以插入日志\" ON operation_logs
  FOR INSERT WITH CHECK (true);
```

## 三、现有表的 RLS 策略补充

客服网页需要对现有表做更多操作（如修改报账、管理车辆等），需要补充以下 RLS 策略：

```sql
-- expense_records 表：客服可以修改待确认的记录（填提成、修改费用等）
-- 现有策略已允许 status='pending' 的 UPDATE，无需额外添加

-- vehicles 表：客服可以增删改车辆
CREATE POLICY \"客服可以插入车辆\" ON vehicles
  FOR INSERT WITH CHECK (true);

CREATE POLICY \"客服可以修改车辆\" ON vehicles
  FOR UPDATE USING (true);

-- drivers 表：客服可以修改司机信息
CREATE POLICY \"客服可以修改司机信息\" ON drivers
  FOR UPDATE USING (true);

CREATE POLICY \"客服可以插入司机\" ON drivers
  FOR INSERT WITH CHECK (true);

-- advance_fund_records 表：客服可以插入充值记录
CREATE POLICY \"客服可以插入备用金记录\" ON advance_fund_records
  FOR INSERT WITH CHECK (true);

CREATE POLICY \"客服可以修改备用金记录\" ON advance_fund_records
  FOR UPDATE USING (true);

CREATE POLICY \"客服可以删除备用金记录\" ON advance_fund_records
  FOR DELETE USING (true);

-- fee_types 表：客服可以增删改费用类型
CREATE POLICY \"客服可以插入费用类型\" ON fee_types
  FOR INSERT WITH CHECK (true);

CREATE POLICY \"客服可以修改费用类型\" ON fee_types
  FOR UPDATE USING (true);

-- service_staff 表：客服可以查看和管理客服账号
CREATE POLICY \"客服可以查看所有客服信息\" ON service_staff
  FOR SELECT USING (true);

CREATE POLICY \"管理员可以插入客服\" ON service_staff
  FOR INSERT WITH CHECK (true);

CREATE POLICY \"管理员可以修改客服\" ON service_staff
  FOR UPDATE USING (true);
```

## 五、各页面详细需求

### 页面1：登录页

**表单：** 账号 + 密码
**逻辑：** 查 service_staff 表匹配 → 成功存缓存（含 id、name、role），跳转首页 → 失败提示\"账号或密码错误\"
**登录状态保持，除非手动退出。**

### 页面2：数据看板（首页）⭐

客服登录后看到的第一个页面，展示今日和本月的关键数据概览。

#### 顶部信息栏
- 显示当前登录客服姓名、角色
- 当前日期

#### 今日概况卡片组（一行4个卡片）

| 卡片 | 数据来源 |
|------|----------|
| 今日新提交 | 今天 created_at 的 expense_records 数量 |
| 今日待确认 | status=\"pending\" 且 record_date=今天 的数量 |
| 今日已确认 | status=\"confirmed\" 且 confirmed_at 在今天 的数量 |
| 总待确认 | 所有 status=\"pending\" 的数量（不限日期） |

**\"总待确认\"卡片如果数量 > 0，用红色/橙色高亮提醒。**

#### 本月汇总区域

| 显示项 | 计算方式 |
|--------|----------|
| 本月总支出 | 当月所有 confirmed 记录的 total_expense 之和 |
| 本月总提成 | 当月所有 confirmed 记录的 commission 之和 |
| 本月报账条数 | 当月所有 expense_records 数量 |
| 本月加班总天数 | 当月所有司机的加班天数之和（按司机+日期去重） |

#### 各司机本月概览表格

| 司机 | 报账条数 | 支出合计 | 提成合计 | 待确认 | 已确认 | 加班天数 | 备用金余额 |
|------|---------|---------|---------|--------|--------|---------|-----------|
| 徐良斌 | 81 | ¥4039 | ¥3180 | 5 | 76 | 3 | ¥461 |
| ... | | | | | | | |

- 点击司机姓名 → 跳转到报账管理页，自动筛选该司机
- 数据自动刷新，也可手动刷新

### 页面3：报账管理页 ⭐⭐（最核心）

客服日常工作的主要页面，逐条处理司机提交的报账记录。

#### 顶部筛选栏

| 筛选项 | 类型 | 说明 |
|--------|------|------|
| 司机 | 下拉选择（全部/具体司机） | 从 drivers 表读取在职司机 |
| 日期范围 | 日期范围选择器 | 默认当月1日~今天 |
| 状态 | 下拉选择 | 全部 / 待确认 / 已确认 |

**默认显示：全部司机、当月、待确认。**

#### 报账记录列表（表格形式）

| 列 | 说明 |
|----|------|
| 日期 | record_date |
| 司机 | 关联 drivers 表的 name |
| 车牌号 | plate_number |
| 路线/地点 | route |
| 费用明细 | 只显示不为0的费用项，格式如\"停车费50 + 盖章10\" |
| 其他费用 | note_amount + note_detail |
| 支出合计 | total_expense |
| 提成 | commission（待确认时可直接在列表中编辑） |
| 凭证 | 有图片时显示图标，点击可预览 |
| 加班 | is_overtime 为 true 时显示\"加班\"标签 |
| 状态 | pending=橙色\"待确认\" / confirmed=绿色\"已确认\" |
| 操作 | 详见下方 |

#### 操作按钮

**状态为\"待确认\"时：**
- **确认** — 将 status 改为 \"confirmed\"，记录 confirmed_by（当前客服姓名）和 confirmed_at（当前时间），同时写入操作日志
- **编辑** — 打开编辑弹窗（见下方）
- **批量确认** — 勾选多条后一键确认

**状态为\"已确认\"时：**
- **查看** — 只能查看详情，不能修改

#### 编辑弹窗（待确认记录）

点击\"编辑\"后弹出模态框，可修改：

| 可修改字段 | 说明 |
|-----------|------|
| 车牌号 | 搜索式输入，同小程序 |
| 路线/地点 | 文本输入 |
| 各项费用 | 逐项显示，可直接修改金额 |
| 其他费用金额 | note_amount |
| 其他费用说明 | note_detail |
| 提成 | commission，下拉选择：0 / 50（保税区）/ 60（单程）/ 120（来回），也可手动输入 |
| 加班 | is_overtime 开关 |
| 凭证图片 | 可查看已有图片，可追加 |

**修改后自动重新计算 total_expense。**
**保存时更新 updated_at，同时写入操作日志（记录修改了什么）。**

#### 快捷操作：批量填提成

- 勾选多条记录 → 点击\"批量填提成\" → 选择提成金额（50/60/120） → 一次性填入

#### 列表排序

- 默认按 record_date 倒序（最新的在最上面），同一天按 created_at 倒序
- 支持点击列头排序

#### 分页

- 每页50条，底部分页器

### 页面4：总表页

展示所有已确认的报账记录（status=\"confirmed\"），即总表数据。

#### 顶部筛选

| 筛选项 | 说明 |
|--------|------|
| 月份选择 | 默认当月 |
| 司机选择 | 全部 / 具体司机 |

#### 总表展示（表格）

与报账管理页类似，但只显示 confirmed 的记录，且不可编辑。

#### 汇总信息

| 显示项 | 说明 |
|--------|------|
| 筛选范围内总支出 | 所有显示记录的 total_expense 之和 |
| 筛选范围内总提成 | 所有显示记录的 commission 之和 |
| 记录总条数 | — |

#### 导出 Excel ⭐⭐

点击\"导出Excel\"按钮，生成 .xlsx 文件下载。

**导出格式（必须与现有 Excel 模板一致）：**

- **每个司机一个 Sheet**，Sheet 名为司机姓名
- 每个 Sheet 的结构如下：

**第1行（标题行）：**
```
{司机姓名}{年}年{月}月份短驳提成  备用金{该月备用金充值总额}
```

**第2行（表头行）：**
```
日期 | 车牌 | 司机 | 路线 | 过磅费 | 提柜费 | 过夜费 | 越南超时费 | 越南收钥匙 | 停车费 | 新岗 | 打车 | 淋水 | 解篷布 | 高速费 | 盖章 | 备注 | 备注明细 | 支出费用 | 提成 | 图片 | 加班标记 | 日期/客户
```
共23列（A~W列）

**数据行（第3行开始）：**
- 每行 = expense_records 表中该司机当月 status=\"confirmed\" 的一条记录
- 按 record_date 正序排列
- 日期列显示 record_date
- 司机列固定显示该司机姓名
- 费用列：值为0时显示为空（不显示0）
- 备注列：对应 note_amount（其他费用金额）
- 备注明细列：对应 note_detail（其他费用说明）
- 支出费用列：total_expense
- 提成列：commission
- 图片列：如有凭证图片，显示图片URL或标记
- 加班标记列：is_overtime=true 时显示\"加班\"，否则为空

**数据行结束后的汇总行：**
```
（空行后）
支出费用列：所有记录的 total_expense 之和
提成列：所有记录的 commission 之和
加班标记列：显示\"X个加班\"（X = 按 record_date 去重后的加班天数）
```

**底部备用金区域（汇总行下方空2行）：**
```
盖章列          | 备注列
12/31备用金     | 1000
1/8备用金       | 1000
1/15备用金      | 500
...
（空行）
                | 余    | （备用金充值总额 - 支出总额）
```
- 从 advance_fund_records 表读取该司机当月的充值记录
- 余额 = 充值总额 - 支出总额（仅已确认的）

## 六、全局规范

### 响应式设计
- 主要适配电脑浏览器（1280px以上）
- 最低支持1024px宽度
- 不需要适配手机端

### 导航
- 顶部水平导航栏，当前页面高亮
- \"系统设置\"为下拉菜单

### 配色
- 主色：蓝色（#3b82f6）
- 已确认=绿色，待确认=橙色
- 背景=浅灰（#f3f4f6）
- 卡片/表格=白色
- 危险操作（删除等）=红色

### 加载与空状态
- 请求时显示 loading
- 列表空显示\"暂无数据\"
- 网络错误提示\"网络异常，请重试\"

### 数据权限
- 客服可以查看和操作所有司机的数据
- role=\"admin\" 的客服可以访问客服账号管理页
- role=\"staff\" 的客服只能看到除\"客服账号管理\"以外的所有页面

### 编辑权限
- 只有 status=\"pending\" 的报账记录可以编辑和确认
- status=\"confirmed\" 的记录锁定不可修改

### 操作日志
- 以下操作需记录到 operation_logs 表：
  - 确认报账记录
  - 修改报账记录（记录修改了哪些字段）
  - 新增/修改/停用 司机
  - 新增/修改/停用 车辆
  - 新增/修改/删除 备用金充值
  - 新增/修改/停用 费用类型
  - 新增/修改/删除 客服账号

### 操作确认
- 确认报账、删除操作等关键操作需要二次确认弹窗
- 批量确认需显示确认数量

### 汇总行

数据行结束后：
- S列（支出费用）：该司机当月所有已确认记录的 total_expense 之和
- T列（提成）：该司机当月所有已确认记录的 commission 之和
- V列（加班标记）：显示 \"X个加班\"（X = 该司机当月已确认记录中 is_overtime=true 的记录按 record_date 去重后的天数）

### 底部备用金区域

汇总行下方空2行后，在 P列（盖章列）和 Q列（备注列）显示：
- 逐行显示每笔充值：P列 = \"{月/日}备用金\"，Q列 = 金额
- 最后一行（充值记录下方）：S列 = \"余\"，T列 = 备用金余额（充值总额 - 支出总额）

### 标题行

第1行合并显示：
```
{司机姓名}{年}年{月}月份短驳提成  备用金{充值总额}
```

## 八、重要提醒

1. **共用同一个 Supabase 数据库**，不需要创建新的数据库连接，只需新建 H5 网页应用
2. **只需要新建1张表**（operation_logs），其他6张表已存在
3. **总表 = status=\"confirmed\" 的 expense_records 记录**，不是独立的表
4. **提成由客服填写**（确认前），填完提成后再确认
5. **确认后锁定不可修改**，没有\"撤销确认\"功能
6. **备用金余额 = 充值总额 − 已确认支出总额**
7. **加班天数统一规则**：expense_records 中 status=\"confirmed\" 且 is_overtime=true 的记录，按 record_date 去重后的天数
8. **Excel 导出格式必须与现有模板完全一致**（参见第七节），包括列顺序、标题格式、底部备用金区域
9. **操作日志只记录客服的操作**，司机在小程序端的操作不记录到此表
10. **技术栈**：与小程序相同 — Taro + React + TypeScript + Supabase + Tailwind CSS，但客服后台为 H5 网页应用（电脑浏览器访问）
